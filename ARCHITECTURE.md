# Архитектура

Лента коротких видео на чистом JavaScript: ES-модули через import map, ноль
runtime-зависимостей, запуск с любого статического сервера. Каждое решение ниже
выводится из критериев задания - **структура кода, отзывчивость, эффективность
использования ресурсов браузера** - и объяснено через «почему», а не только «как».

Отправная точка: лента - это задача про три дефицитных ресурса с разной ценой.

| Ресурс | Лимит | Наш ответ |
|---|---|---|
| DOM-узлы | память (мягкий) | N дешёвых пустых ячеек + 9 переиспользуемых слайдов |
| Видеодекодеры | железо: 1-2 на мобильных (жёсткий) | максимум 2 живых `<video>`: активный + прогретый |
| Сеть | общий канал | байты качает только активный (+метаданные соседа) |

## 1. Два сложных места

### 1.1 Виртуализация, совместимая с нативным scroll-snap

**Проблема.** Показать сотни видео без сотен узлов в DOM - нужна виртуализация.
Но наивный вариант (повесить `scroll-snap-align` на ~9 переиспользуемых слайдов)
ломается: точки прилипания браузер берёт с реальных элементов, и быстрое
инерционное листание может остановиться там, где слайда сейчас нет - снап
теряет цель, прокрутка дёргается или зависает между экранами.

**Решение - разделить сетку прилипания и слой отрисовки:**

```
#feed  (scroll-контейнер: overflow-y:scroll; scroll-snap-type:y mandatory)
│
├── .cell × N        пустые div'ы в потоке, height:H, scroll-snap-align:start
│                    = ПОСТОЯННАЯ сетка снап-точек + высота скроллбара.
│                    Дыр нет: прилипнуть можно к любому индексу всегда.
│
└── .slide × 9       тяжёлые узлы (постер, видео, оверлей), position:absolute,
                     transform: translate3d(0, index·H, 0), scroll-snap-align:none
                     - едут поверх ячеек, в прилипании не участвуют.
```

Почему это работает быстро: слайды лежат *внутри* прокручиваемого контейнера,
поэтому во время прокрутки их двигает сам браузер на уровне композитора - **JS в
прокрутке не участвует вообще**. Переиспользование (переезд дальнего слайда на
новый индекс) происходит только при смене активного индекса, вне экрана.

**Почему вся геометрия в пикселях, а не в `dvh`.** Высота мобильного вьюпорта
«дышит» (адресная строка прячется). Если ячейки посчитаны в CSS-единицах, а
трансформы слайдов в JS-пикселях, они разъезжаются и снап дрожит. Единственный
источник истины - `core/viewport.js`: измеряет `H = clientHeight` и публикует
её и в CSS-переменную `--slide-h` (ячейки), и в JS (трансформы). Смена высоты →
всё пересчитывается из одного числа.

**Активный слайд определяется без опроса `scrollTop`.** Чтение геометрии в
обработчике `scroll` - форсированный layout посреди кадра (layout thrashing).
Вместо этого `IntersectionObserver` с центральной полосой
(`rootMargin: '-50% 0px -50% 0px'`) сам сообщает, какая ячейка пересекла центр
экрана. Финальное решение «играем вот это» принимается один раз на остановке
(`scrollend` + таймер-фолбэк 140 мс) - быстрое листание через 8 роликов
поднимает один декодер в точке назначения, а не восемь по дороге.

### 1.2 Бюджет декодеров ≠ бюджет DOM

Мобильные SoC имеют 1-2 аппаратных видеодекодера. Превышение лимита в Chrome -
`Blocked attempt to create a WebMediaPlayer`, на iOS - молчаливое голодание.
Поэтому «слайд в DOM» (9 шт.) и «слайд держит декодер» (≤2) - независимые
понятия: 7 слайдов из 9 показывают постер `<img>` (ноль декодеров, ноль сети).

**Обязательная последовательность освобождения** (иначе декодер и сетевая
загрузка утекают - `pause()` сам по себе их НЕ освобождает):

```js
video.pause();
video.removeAttribute('src');   // отвязать источник (не src='' - та резолвится в URL страницы!)
video.load();                   // перезапуск load-алгоритма: рвёт fetch, отдаёт декодер
```

**Поведение при листании.** Шаг на соседа - `pause()` (декодер жив: вернёшься -
мгновенный resume). Прыжок дальше соседа - `suspend()`: все декодеры
освобождаются, пролёт «мостят» постеры, декодирование поднимается один раз на
остановке. Прогреваемый сосед выбирается **по направлению движения** (листаешь
вверх - греется `index-1`). Пул - LRU на порядке вставки `Map`, ёмкость задаётся
одной константой в `main.js`; число прогреваемых выводится из неё (`capacity-1`).

**Политика автозапуска.** Браузеры запрещают автоплей со звуком без жеста
пользователя, поэтому: `muted` (атрибутом, до установки `src`) + `playsinline` +
старт всегда без звука; глобальный флаг звука живёт в одном владельце и
наследуется каждым следующим роликом. Отказы `play()` - не ошибки:
`AbortError` - штатная гонка листания, `NotAllowedError` - «жди жеста»; оба
осознанно проглатываются.

## 2. Слои и связи

```
core → engine → data / media / view / input   - слои не лезут друг в друга
                 └── связывает всё FeedController (медиатор) через EventBus
```

| Слой | Модули | Зона ответственности |
|---|---|---|
| core | EventBus, RafScheduler, viewport | шина событий; единственный rAF-цикл (фазы чтение→запись); измеренная H |
| engine | WindowCalculator, Virtualizer | виртуализация; **ничего не знает про видео** |
| data | manifest, FeedService, LikesStore | контракт имён файлов; постраничный клиент; лайки |
| media | VideoPool, PlaybackController | пул декодеров; политика воспроизведения |
| view | SlideView, ActionsOverlay | пассивные вью: «покажи/скрой», ноль логики |
| input | GestureController, KeyboardController | жесты/клавиши → семантические интенты |

**Почему медиатор и события, а не прямые вызовы.** Прямые ссылки дали бы граф
«все со всеми» (Virtualizer знает про PlaybackController, тот - про SlideView…).
Вместо этого модули публикуют факты (`slide:enter`, `feed:page`, `intent`), а
сценарий целиком знает один `FeedController` - читается в одном файле,
тестируется каждый модуль в изоляции. Словарь событий типизирован
(`src/events.js` + JSDoc-дженерик шины): переименование поля payload ловит
`tsc --checkJs`.

**Почему один rAF-цикл.** `RafScheduler` выполняет в кадре сначала все чтения
DOM, потом все записи - layout thrashing невозможен конструктивно, а не по
договорённости. Прямой вызов `requestAnimationFrame` в модулях запрещён.

**Прочие инварианты:** анимации только композитные (`transform`/`opacity` -
прогресс-бар это `scaleX`); обработчики только делегированные (слайды
переиспользуются - персональные листенеры утекли бы); teardown полный
(подписки/обсерверы/таймеры), страница дружит с bfcache.

## 3. Поток данных

```
свайп → нативный scroll (0 JS) → IntersectionObserver: «кандидат N»
  → окно слайдов едет за пальцем (постеры), исходящий звук на паузу
  → остановка (scrollend/таймер) → commit: страница данных? окно? декодер? play
  → приближение к концу загруженного → FeedService тянет следующую страницу
```

Рост ленты только append-only: `scrollTop` никогда не прыгает.

## 4. Пайплайн данных

Источник - публичная папка Google Drive. Стримить из неё напрямую нельзя
(нестабильные Range-запросы, квоты, CORS) - демо у проверяющего зависало бы.
Поэтому медиа обрабатывается **один раз на этапе сборки** (`npm run build:feed`)
и коммитится в репозиторий - клонировал и запустил, без внешних зависимостей:

1. скачивание папки (`gdown`), кэш исходников вне git;
2. нормализация: H.264/`yuv420p` (декодится везде), 960p/crf27 (лёгкий репозиторий),
   `+faststart` - метаданные в начале файла, воспроизведение с первого запроса;
3. постер = первый кадр (ffmpeg), тот же размер, что видео - стык постер→кадр незаметен;
4. постраничный JSON-манифест. Имена файлов страниц - общий модуль
   `src/data/manifest.js`, его импортируют и генератор, и рантайм-клиент:
   производитель и потребитель не могут разойтись.

Приложение потребляет манифест как настоящий постраничный API (ленивая загрузка,
дедупликация запросов, префетч следующей страницы) - замена на реальный бэкенд
сводится к подмене `baseUrl`.

## 5. Чеклист производительности

- один rAF-цикл, фазы чтение→запись; `scrollTop` не опрашивается в прокрутке;
- анимации только `transform`/`opacity`; `will-change` - только на 9 слайдах;
- геометрия из одного измеренного числа `H` (px), не из `dvh`;
- декодеры: ≤2, при пролёте 0; освобождение декодера = освобождение сети;
- `modulepreload` всего графа модулей + preload первых JSON - минус каскад загрузок;
- прогресс-бар пишет `transform` только при видимом изменении (квантование);
- постеры отменяют чёрные вспышки и дают быстрый LCP.

**Замерено** (Chrome DevTools, см. README): LCP 198 мс, CLS 0.00, INP 48 мс
(JS - 2.6 мс), куча ~2 МБ / 1 GC-сборка на 10 с листания, 0 ошибок консоли.

## 6. Риски и фолбэки

| Риск | Ответ |
|---|---|
| `scrollend` не везде поддержан | таймер тишины 140 мс как основной путь |
| Снап на transform-элементах глючит в старых iOS | снап на ячейках - слайды в нём не участвуют |
| `play()` отклоняется | `AbortError`/`NotAllowedError` - состояния, не ошибки |
| Возврат из bfcache | на `pagehide` только паркуем декодеры; `visibilitychange` реанимирует |
| Drive недоступен при сборке | медиа уже в репозитории; `--local` собирает из кэша |
